# brochure

**Purpose**: Defines the HTML structure and section-slot contract for all 12 static content sections of the Arangetram brochure webpage. Controls section ordering, exposes stable section IDs consumed by the i18n and print modules, and provides the content skeleton the owner fills manually. (serves FR-1, FR-2, FR-3, FR-4, FR-5, FR-6, FR-7, NFR-1, NFR-2)

**Public surface**

Section IDs — stable anchors consumed by i18n (content swapping) and print (visibility/page-break rules):
- `#cover` — Page 1: title, performer name, guru name, event date/time/venue
- `#school-guru` — Page 2: school mission text + guru's personal message with photo
- `#maargam` — Pages 3–9: container for all 7 dance items
- `#dance-item-1` through `#dance-item-7` — individual dance items; each contains slots for title, Raagam, Thaalam, Composer, Language, Choreography credit, description paragraph, and dancer photo
- `#artist-bio` — performer biography: photo + multi-paragraph bio text
- `#guru-bio` — guru biography: photo + bio text
- `#orchestra` — orchestra bios: one sub-section per musician (vocalist, violinist, mridangist), each with photo + bio
- `#gratitude` — credits list in role: name format
- `#feedback-mount` — empty `<div>` at the bottom of the page; the feedback module mounts `#feedback-section` here
- `#i18n-toggle-mount` — empty `<div>` in the page header; the i18n module mounts the language toggle button here
- `#print-trigger-mount` — empty `<div>` in the page header; the print module mounts the print button here

Content slot convention — all translatable text elements carry a `data-i18n="<key>"` attribute using the `<sectionId>.<fieldName>` pattern:
- Example: `data-i18n="cover.performerName"`, `data-i18n="dance-item-1.raagam"`, `data-i18n="artist-bio.para1"`
- The brochure owns the attribute name (`data-i18n`) and key pattern; the i18n module owns the lookup table and swap logic.
- Images carry `data-i18n-alt="<key>"` for translatable alt text.

**Invariants**
- Section order is managed by the visual editor (FR-17); reordering via the editor is expected and not a hard-flag change. Section IDs are stable regardless of order — i18n and print CSS selectors remain valid after any reorder.
- Every section is a `<section>` element with the exact `id` listed above. Renaming any ID is a hard-flag change (i18n and print both depend on them).
- All translatable text elements carry `data-i18n` attributes — brochure does not manipulate their content at runtime.
- Images live under `assets/images/`; filename conventions are left to the owner.
- The brochure module contains zero JavaScript — it is pure HTML structure and static CSS only.
- Mount points (`#feedback-mount`, `#i18n-toggle-mount`, `#print-trigger-mount`) are always present in the DOM; peer modules never create their own insertion points.

**Key choices**
- `[D-004]` Stable section IDs as the cross-module contract — section IDs are the single source of truth for i18n targeting and print CSS selectors. Renaming requires coordinated changes across three modules.
- `data-i18n` attribute convention defined here (not in i18n) so the brochure's HTML is self-describing — a reader can see which elements are translatable without loading the i18n module.

**Non-goals**
- No dynamic content loading or server-side templating.
- No routing or URL-based navigation — section links are plain `#anchor` hrefs.
- Does not own the feedback form or list (owned by `feedback`; brochure only provides `#feedback-mount`).
- Does not own the language toggle (owned by `i18n`; brochure provides `#i18n-toggle-mount`).
- Does not own the print trigger button (owned by `print`; brochure provides `#print-trigger-mount`).
- No CMS or database integration.
