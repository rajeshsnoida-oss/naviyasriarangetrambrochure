# editor

**Purpose**: Provides a local-only visual authoring interface for the brochure, allowing the owner to edit text content, swap images, reorder sections, add content cards within existing sections, and adjust design tokens without editing source files manually. On save, writes directly to `index.html` and `css/style.css`. Not deployed to GitHub Pages. (serves FR-14, FR-15, FR-16, FR-17, FR-18, FR-19, FR-20, NFR-5)

**Public surface**

- Local dev server URL — `localhost:<port>/editor`; started via a zero-install static server (e.g., `npx serve` or `npx live-server`)
- Save action — writes serialised HTML to `index.html` and updated CSS custom property values to the `:root` block of `css/style.css`
- No exported JavaScript functions — the editor exposes no programmatic API to peer modules

**Invariants**
- The editor is never deployed or referenced by the GitHub Pages build
- Saved HTML is valid and deployable to GitHub Pages without any additional transformation
- All brochure section IDs (`#cover`, `#school-guru`, etc.) are preserved in saved HTML regardless of reorder operations, so i18n, print, and feedback modules remain functional
- `data-i18n` and `data-i18n-alt` attributes are preserved as pass-through in saved HTML; the editor does not read or write their values
- Style changes are written exclusively as CSS custom property values in the `:root` block — no inline styles are injected into HTML
- Tamil content is not displayed or edited; Tamil strings are managed directly in source files

**Key choices**
- `[D-009]` Option A — editor serialises DOM to static HTML; no content data layer; deployed brochure remains pure static HTML; satisfies FR-20 with no build step
- `[D-010]` GrapesJS as the editor framework — open-source WYSIWYG that serialises to HTML/CSS; natively supports drag-reorder, style manager, image swap, and block-level components

**Non-goals**
- No Tamil content authoring (per NFR-5)
- No deployment to GitHub Pages
- No GitHub API integration or auto-commit
- No multi-user editing or authentication
- No undo history beyond a single browser session
- Does not own section structure — IDs, slot conventions, and data-i18n patterns are owned by the brochure module
